(Dev Toolchain) Game

An dodge-the-fire balloon game created using libraries via web development toolchain

Overview and Usage

The core functionality of this project is that in the game, users control a blue balloon. Using the arrow controller keys on their keyboard, users can move the balloon left, right, up, and down at a set speed. If the balloon moves off the page, it reappears on the other side (left to right side of screen, top to bottom of screen, and vice versa) to keep the player in the game. Meanwhile, fireballs rotating at different speeds move across the screen, which the user has to avoid. If the user's balloon gets hit by a fireball, then the balloon expands. If the score reaches 35 hits, the game is over.

Smaller functionalities included in this game are that the background is responsive. As the balloon acquires more hits and expands, the background red arches further shift to the right side of the screen. The game includes instructions in the center bottom of the screen to guide users at the beginning of the game and disappears after the user begins the game and has a score of 2 hits. Additionally, another functionality is that users can click anywhere on the screen to play or pause the game background music. By default, the music is off. The sound icon in the top right corner updates to show which sound state is selected.

Development Process

The steps I took for this project included:

  1. Following an in-class Dev Toolchain Walkthrough activity to install libraries.
  2. Reading up on P5.play functions through the P5.play library.
  3. Going through examples in Allison Parrish's "Making Games with p5.play" tutorials and observing which P5.play functions they leveraged to create their interactions and animations. In the Kudos section, I specify which examples I studied in particular.
  4. Building out the most basic functionality of the game and then working my way up to building the more complex attributes. This involved first creating a sprite for the player and having it controlled by the arrow keys. I then added the fireballs and defined their moving behavior, wrote the code to make the balloon expand when it overlaps with the fireballs, and added background effects such as the background arcs moving right as the number of hits increase. I then added visual elements such as the score number of hits at the top of the screen, and then a control for sound. I focused on working on the elements involving p5 and p5.play first, and then worked on adding the sound elements with p5.sound.
  5. Debugging errors that occurred along the way via the console. I attended office hours for errors I had trouble solving on my own.
  6. Revisiting the in-class activity again to figure out how to set up the live server to serve my code and bundle it with Rollup. This included getting help via Discord to double check my understanding of instructions.
  7. Following an example game provided by the professor to double check my process for bundling my code up with Rollup.
  8. Deploying the game to Github pages.

Issue Deep-Dive

One logic error I ran into occurred when I defined the keyboard controls of the balloon player. When using the right arrow key to attempt to move the balloon to the right, the balloon's position would not update. Using console.log, I found that the number for keycode (which distinguishes between which key controls are selected) wasn't updating for the right arrow after the first move was made. I found that the error was occurring because the balloon's direction and speed were being set together, rather than separately. Instead of using setSpeed, I resolved the issue by defining the player speed and direction using separate functions. This was an error that I received help in my interpretation of its meaning during office hours. What I learned along the way was how to better interpret the error messages provided within the console to narrow down what the source of the problem might be.

Ideas and Future Work

Some ideas I might consider implementing in the future include:

  • Allowing users to choose the game difficulty level. This would be controlled by the speed of the fireballs, how many fireballs there are, the size of the fireballs, the speed of the balloon, etc.
  • Providing player customization: users being able to choose other player avatars besides a balloon. Alternatively, this could entail users being able to specify the color of the balloon.
  • Allowing users to customize the game setting by being able to choose between a selection of game backgrounds.
  • Implementing a life system and incrementing the game's difficulty across several stages for a larger game journey.
